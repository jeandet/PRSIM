Where does the Window abstraction live?

The window needs to be a concept in core, but backends create the actual OS window. Who owns what?

A

Core owns Window, Backend implements it

Core defines a Window interface (abstract class). Each backend provides its own implementation (SDL window, headless window, etc.). The app creates windows through a factory on the backend.

Pros

  • Clean separation — core code can hold Window references
  • Natural for multi-window: each Window has its own WidgetTree
  • Backend polymorphism already established

Cons

  • Virtual dispatch on window operations
  • Backend must expose a window factory method
B

Window is a value type in core, Backend maps it

Core defines WindowConfig (value type with properties) and WindowId (handle). The backend creates OS windows from configs and maps IDs internally. Core never touches the OS window directly.

Pros

  • No virtual dispatch — WindowId is just an integer
  • Matches the existing pattern (BackendConfig is already a value type)
  • Easy to serialize/send across threads
  • Headless backend just tracks configs, no real windows

Cons

  • Core can't call window methods directly — must go through backend
  • More indirection for operations like set_title()
C

Hybrid: WindowHandle in core, owned by Backend

Core defines a thin WindowHandle with non-virtual inline methods that forward to the backend via a stored pointer. The backend creates and owns the actual window objects. The handle is a lightweight view.

Pros

  • Ergonomic API: window.set_title("foo")
  • No virtual dispatch on the handle itself
  • Backend owns lifetime, handle is just a view

Cons

  • Dangling handle risk if backend destroys window
  • More complex than pure value types
