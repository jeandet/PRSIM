Plot Widget — Architecture Approaches

How to integrate the plot into PRISM's MVB architecture

A

Pure Canvas Component

A model struct with canvas() — everything drawn in one DrawList. Axes, grid, data, cursor all in a single draw call. Simplest, follows the dashboard waveform pattern exactly.

Pros

  • Minimal new abstractions
  • Proven pattern (dashboard already does this)
  • One file, one struct

Cons

  • All rendering logic in one function — gets big
  • No reusable axis/grid/legend primitives
  • Cursor overlay redraws entire plot
B

Layered Canvas Component

Same canvas() approach, but internally decomposed into layers: background (grid), data (series polylines), overlay (cursor/crosshair). Each layer is a pure function (DrawList&, Rect, params) → void. The canvas() call orchestrates them.

Pros

  • Clean separation without new framework concepts
  • Layer functions are testable in isolation
  • Same proven canvas pattern externally
  • Layers can be cached/skipped independently later

Cons

  • Still redraws everything on cursor move (for now)
  • Slightly more internal structure
C

Sentinel Widget (Delegate<Plot>)

A new Plot sentinel type with a dedicated Delegate<Plot> specialization. The plot becomes a first-class widget type like Slider or Table, with its own record() and handle_input().

Pros

  • First-class framework citizen
  • Consistent with Slider/Table pattern
  • Can leverage WidgetNode state (hover, focus)

Cons

  • More boilerplate (Delegate specialization)
  • Sentinel pattern designed for single-value fields, not multi-series data
  • Harder to compose — plot config doesn't fit Field<T> cleanly

Recommendation: B — Layered Canvas

It follows the proven canvas pattern (no new framework concepts), keeps the rendering logic clean via pure layer functions, and leaves the door open for per-layer caching when GPU backends arrive in Phase 5. The sentinel approach (C) fights the MVB grain — plots are complex compositions, not single-value widgets.
